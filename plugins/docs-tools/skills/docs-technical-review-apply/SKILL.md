# Technical Review Apply Skill

Apply fixes from technical review reports interactively using an agentic workflow that presents each issue with context and gets user approval.

## Purpose

This skill implements **Phase 2** of the technical review workflow, handling issues that require human judgment (<65% confidence). It:
- Reads technical review reports generated by `docs-technical-review-validate`
- Filters items by confidence range or specific IDs
- Presents each fix with full context (code evidence, reasoning, suggested changes)
- Uses `AskUserQuestion` to get approval, modification, or skip decisions
- Applies approved changes using content-based matching and tracks results
- Updates the report with applied fixes

## Usage

### Apply All Items Below Confidence Threshold

```bash
# Apply all medium-confidence items (50-64%)
/docs-technical-review-apply .claude_docs/technical-review-report.md --confidence 50-64

# Apply all low-confidence items (requires more judgment)
/docs-technical-review-apply .claude_docs/technical-review-report.md --confidence 0-49
```

### Apply Specific Items by ID

```bash
# Apply selected manual review items
/docs-technical-review-apply .claude_docs/technical-review-report.md --items MR-1,MR-3,MR-5

# Apply single item
/docs-technical-review-apply .claude_docs/technical-review-report.md --items MR-1
```

### Apply All Manual Review Items

```bash
# Apply everything in the manual review section
/docs-technical-review-apply .claude_docs/technical-review-report.md
```

## Interactive Workflow

For each item to review, the skill:

1. **Load item from report** - Read item details from JSON sidecar (or parse markdown fallback)
2. **Read current file** - Get the actual content using the `old_text` field for context
3. **Format for presentation** - Create clear before/after comparison
4. **Use AskUserQuestion** - Present options:
   - Apply suggested fix
   - Modify fix (ask for changes)
   - Skip (leave for manual editing)
   - Delete section entirely (if appropriate)
5. **Apply decision** - Use `Edit(file_path=FILE, old_string=old_text, new_string=fix)` for content-based matching
6. **Track results** - Record what was applied, skipped, or modified
7. **Update report** - Mark applied items in the report

## Implementation Workflow

### Step 1: Parse Arguments

Extract the report file path (first positional argument) and optional filters:
- `--confidence MIN-MAX` — only process items in this confidence range
- `--items ID,ID,...` — only process these specific item IDs
- `--dry-run` — preview items without applying changes

### Step 2: Load Items from JSON Sidecar

Read the JSON sidecar (same path as the report but with `.json` extension). Filter for items with confidence <65%, then apply any `--confidence` range or `--items` ID filters. If the JSON sidecar is missing, fall back to parsing the markdown report (warn the user that content matching may be less precise).

Each JSON item provides these fields:

| Field | Description |
|-------|-------------|
| `id` | Item identifier (e.g., `MR-1`) |
| `file` | Path to the documentation file |
| `line` | Line number in the file (approximate — use `old_text` for matching) |
| `category` | One of: `commands`, `code_blocks`, `apis`, `configs`, `file_paths`, `conceptual` |
| `old_text` | Exact current text in the file to match against |
| `new_text` | Suggested replacement text |
| `confidence` | Confidence score (0-100) |
| `evidence` | Code evidence string |
| `description` | Human-readable issue description |
| `severity` | Issue severity (`High`, `Medium`, `Low`) |
| `reasoning` | Explanation of why the change is suggested |

### Step 3: Process Each Item Interactively

For each item, present the issue with context and get user approval:

1. **Read current file** to verify `old_text` exists in the file
2. **Present** the item in this format:

```
MR-1 (1 of 5): Command flag renamed | Confidence: 60% | Severity: High
File: modules/proc-install.adoc

Current:  $ my-tool --enable-feature
Suggested: $ my-tool --feature-enable

Evidence: Flag renamed in commit abc123, git log confirms deprecation
Reasoning: Exact command exists, only the flag changed — likely a rename
```

3. **Ask user** via `AskUserQuestion` for their decision
4. **Apply fix** using content-based matching:

```bash
# Apply using Edit tool with content-based matching (no line numbers needed)
Edit(file_path=FILE, old_string=old_text, new_string=new_text)
```

This approach uses exact text matching rather than line numbers, so edits to earlier parts of the file do not cause fixes to be applied to the wrong location.

### Step 4: Display Summary and Update Report

Display counts of applied, modified, skipped, and deleted items. If any changes were made, back up the report file and mark applied items in the markdown report by prefixing their headers with "APPLIED".

## User Interaction

For each item, use `AskUserQuestion` to present four options:
1. **Apply suggested fix** — use the fix as-is
2. **Modify fix** — follow up to ask what the user wants changed, then apply the modified version
3. **Skip** — leave for manual editing later
4. **Delete section** — remove the entire section from documentation

If the user chooses "Modify fix", ask them to describe the changes they want, apply the modification, and confirm with them before moving to the next item.

## Error Handling

| Error | Handling |
|-------|----------|
| Report file not found | Exit with error message |
| JSON sidecar missing | Fall back to markdown parsing with warning |
| Invalid report format | Exit with error message |
| No items match filters | Exit gracefully with message |
| File not found for fix | Skip item, report error, continue |
| `old_text` not found in file | Warn user, show current file content, ask how to proceed |
| Edit operation fails | Report error, ask to retry or skip |

## Integration

This skill is designed to work with:
- `docs-technical-review-validate` - Generates the report and JSON sidecar this skill reads
- `docs-workflow` - Could be integrated as post-review step
- CI/CD - Could be used for semi-automated fixing
